Here is a list of the common problems from the document:

* **Inconsistent Data Types & Mismatches:** Numeric fields are often inconsistently stored. There are also data type mismatches where data is returned in a different type than defined in the schema (e.g., returning a string `"28"` instead of an integer `28`), or non-deterministic types like `Union[int, float]` are used. Explicit typecasting to the actual schema datatype is required before saving or returning data.
* **Incorrect Tool Description Formatting & Grammar:** Tool descriptions must be wrapped in double-quotes (`""`) instead of parentheses (`()`). Additionally, descriptions must use the third-person singular form (e.g., "Creates a workspace" instead of "Create a workspace").
* **Parameter Validation Issues:** Required parameters sometimes lack implementation validation (they must return an error if missing and be present in the `get_info` "required" array). Conversely, fully optional parameters are improperly validated or incorrectly placed in the "required" array. Conditional requirements (like "at least one" or "exactly one") are also often ignored.
* **Missing or Misplaced "Required" Arrays and Labels:** When all fields are optional, the "required" array is sometimes omitted entirely instead of being listed as an empty list (`[]`). Furthermore, "required" and "Optional" labels are improperly placed in the `get_info` description or `invoke` docstring rather than in the specific field information.
* **Missing Field Descriptions and Default Values:** Fields listed in the `get_info` method frequently lack their respective "description" components and fail to include default values.
* **Overly Long or Unclear Function Descriptions:** Tool `get_info` descriptions often exceed 1 or 2 lines or include unnecessary argument details. Vague descriptions can confuse the model during tool selection. It should not show the tool implementation but abstracts it and says when to use the function instead based on the policy.
* **Incorrect Timestamps and Dates:** Using `datetime.now()` instead of a mutually decided timestamp for `created_at` and `updated_at` fields, and using date formats that do not match the database format.
* **Scope and Code Placement Errors:** Functions, constants, or unused imports are left outside the `Tau` class instead of keeping everything properly scoped within the `invoke` method. Extraneous docstrings are also improperly added to the classname or the `invoke` function.
* **Incorrect Return Formats & Input/Output Mismatches:** Failing to return data in the `json.dumps()` format inside the `invoke` function, or having input and output formats that do not match (e.g., taking an integer input and returning a float output).
* **Missing Enum Keys & Status Information:** Failing to include the `enum` key in `get_info` for fields that act as enums, and failing to list the available statuses for tools designed to change a status.
* **Improper Type Hinting for Optionals:** Incorrectly using `Optional[type] = None` when the field actually has a different default value.
* **Non-Matching Error Messages:** Error messages fail to correctly match the specific error being presented.
* **Not Using Domain-Specific Terms:** Using incorrect terminology in function signatures and responses instead of domain-specific terms.
